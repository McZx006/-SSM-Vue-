# 项目来源与二次完善说明

## 说明目的

本说明用于交代项目的选型思路、改造依据和当前成果边界。用户最初希望参考 GitHub 上的个人记账看板项目继续完善，使其成为一个功能较完整、界面友好、后续可以迁移为微信小程序的小型项目。

在当前环境中，没有检索到一个可以稳定引用、且同时满足 SSM、Vue、ECharts、MySQL、RESTful 和课程要求的现成 GitHub 仓库。因此本项目采用更稳妥的方式：以当前已有的 SSM + Vue 个人记账看板为基础，按课程要求和小程序化体验目标进行二次完善。

## 选型标准

如果后续需要替换或合并 GitHub 项目，建议优先选择符合以下条件的仓库：

- 后端使用 Spring、SpringMVC、MyBatis，或者结构接近 SSM，便于课程答辩说明。
- 前端使用 Vue、Axios、ECharts，页面可以清晰展示图表和表单交互。
- 数据库使用 MySQL，至少包含用户、账单、分类三类核心数据。
- 接口风格接近 RESTful，便于和前端、小程序端共用。
- 功能范围适中，适合课程项目展示，不依赖复杂云服务。
- 代码结构清楚，便于二次开发、调试和答辩讲解。

## 当前基线

当前项目已经具备完整的 Web 端和后端基础：

- 后端：SSM 架构，包含 Controller、Service、Mapper、实体类和 MyBatis XML。
- 前端：Vite + Vue 3 + Element Plus + Axios + ECharts。
- 数据库：MySQL，包含用户表、账单表、分类表、预算表等。
- 部署：Maven 打包为 WAR，通过 Tomcat 运行，前端通过 Vite 开发服务器访问。

因此，当前项目比“重新寻找并套用一个不确定仓库”更适合作为课程交付版本。这样可以保证运行路径、接口路径、数据库脚本和页面逻辑都能被完整解释。

## 二次完善内容

本次完善的重点不是简单改样式，而是把项目从“能跑的记账页面”提升为“可演示、可答辩、可扩展的小型产品原型”。

已完成的主要改造包括：

- 登录注册界面产品化：增加视觉引导、体验账号入口、密码强度提示、表单校验和错误提示。
- 主界面移动端化：保留桌面侧边栏，同时为小屏幕提供底部 Tab 导航和浮动记账按钮。
- 仪表板增强：展示本月收入、本月支出、本月结余、预算进度、支出分类饼图、月度收支柱状图和智能建议。
- 账单管理增强：支持新增、编辑、删除、分页、条件筛选、快捷筛选和当前页收支合计。
- 统计分析增强：支持分类占比、分类排行、年度趋势折线图和消费洞察。
- 系统设置增强：支持个人资料、预算、分类维护、深色模式、紧凑模式和 Excel 导出。
- 数据体验增强：补充中文分类、中文备注、2025/2026 演示数据和旧数据中文化脚本。
- 小程序方向增强：新增 `miniapp-prototype` 原生微信小程序页面骨架，并接入登录、注册、首页摘要、分类、记账、账单列表和统计接口，保留后端 RESTful 接口复用路径。
- 文档完善：补充运行说明、交付说明、答辩指南、接口文档、数据库设计、代码导读和小程序改造方案。

## 小程序化改造思路

当前 Web 端已经按照“移动优先”的方向做了体验调整，后续迁移微信小程序时可以复用后端接口和业务逻辑。

推荐迁移路线：

1. 保留 SSM 后端接口，统一返回 JSON。
2. 将 Session 登录逐步改造为 Token 登录，方便小程序端保存登录态。
3. 将 Web 页面拆成小程序页面：首页、记一笔、账单、统计、我的。
4. 图表部分使用 ECharts 小程序适配方案。
5. 将本地开发地址替换为 HTTPS 域名，并在微信公众平台配置合法请求域名。

项目中已经提供 `miniapp-prototype`，用于展示页面结构和接口调用方式。

## 后续如果提供 GitHub 仓库

如果后续指定一个真实 GitHub 仓库，可以按以下方式继续合并：

- 先比较技术栈是否和当前 SSM + Vue 项目一致。
- 如果对方后端更完整，可迁移其接口设计和数据库字段。
- 如果对方前端体验更好，可迁移其页面布局、交互和图表组织方式。
- 当前项目中的登录、账单、统计、设置、小程序原型和文档可以作为保留成果。
- 合并前优先保证数据库脚本、接口路径和运行说明不被破坏。
