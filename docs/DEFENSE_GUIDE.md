# 答辩与演示指南

## 1. 一分钟项目介绍

本项目是一个面向大学生日常消费场景的个人记账看板。传统记账只是记录流水，用户很难直观看出钱花在哪里、支出是否超预算、哪几个月消费波动比较明显。因此项目使用 SSM 搭建后端接口，使用 Vue、Axios、ECharts 搭建前端页面，把账单数据转化为饼图、柱状图、折线图和智能建议。

项目不仅实现了 Web 端完整功能，也提供了微信小程序原型目录，方便后续扩展到移动端。

## 2. 演示顺序

建议演示控制在 5 到 8 分钟。

1. 打开登录页，使用测试账号 `testuser / 123456` 登录。
2. 展示仪表板：本月收入、本月支出、结余、预算使用率、分类饼图、月度柱状图、智能建议。
3. 点击右下角“记一笔”，使用早餐或通勤模板快速新增账单。
4. 观察仪表板或账单页数据刷新，说明通过 `bill-created` 事件触发页面重新加载。
5. 打开账单管理，演示查询、快捷筛选、编辑和删除。
6. 打开统计分析，演示分类占比、分类排行、月度趋势和智能分析建议。
7. 打开系统设置，演示分类管理、预算设置、深色模式、Excel 导出。
8. 简要展示 `miniapp-prototype`，说明小程序原型包含登录、注册、记一笔、账单列表等页面。

## 3. 重点功能讲解

### 登录注册

讲解点：

- 前端使用 `Login.vue` 和 `Register.vue`。
- 用户状态由 Pinia 的 `stores/user.js` 管理。
- 后端登录接口是 `POST /api/auth/login`。
- 登录成功后，后端把用户放入 Session。
- `AuthInterceptor` 会拦截需要登录的接口。

可以这样说：

```text
本系统当前使用 Session 作为后端鉴权方式，前端 localStorage 主要用于保存用户展示状态和路由守卫。正式小程序上线时，可以进一步升级为 token 鉴权。
```

### 账单管理

讲解点：

- 核心表是 `bill`。
- 每条账单包含金额、类型、分类、日期、备注和用户 ID。
- 前端页面是 `Bills.vue`。
- 后端接口是 `/api/bills`。
- MyBatis 在 `BillMapper.xml` 中实现多条件筛选和分页。

可以这样说：

```text
账单查询支持类型、分类、日期范围和关键词筛选，后端通过动态 SQL 拼接条件，并通过 user_id 保证用户只能查看自己的账单。
```

### 图表统计

讲解点：

- 饼图来自 `/api/statistics/category`。
- 折线图和柱状图来自 `/api/statistics/monthly`。
- 仪表板综合接口是 `/api/statistics/dashboard`。
- 后端通过 MyBatis `SUM`、`GROUP BY`、`CASE WHEN` 做统计。
- 前端通过 ECharts 渲染。

可以这样说：

```text
统计不是前端自己算的，而是后端通过 MyBatis 聚合 SQL 计算后返回，前端只负责渲染图表和补齐没有数据的月份。
```

### 快捷记账

讲解点：

- 快捷记账在 `MainLayout.vue`。
- 所有登录后的页面都能使用。
- 支持早餐、通勤、教材、兼职模板。
- 保存成功后触发 `bill-created` 事件。
- 仪表板、账单页、统计页监听该事件并刷新数据。

可以这样说：

```text
这个设计模仿微信小程序的高频入口，让用户不用进入复杂表单，也能快速完成一笔记录。
```

### 智能建议

讲解点：

- 逻辑在 `frontend/src/utils/advice.js`。
- 当前是本地规则版，不依赖外部 AI 服务。
- 根据预算使用率、最高分类、收支比例、年度峰值月份生成建议。
- 可作为后续接入大模型 AI 分析的原型。

可以这样说：

```text
为了保证课程演示稳定，我先实现了本地规则版智能建议。后续如果接入大模型，只需要替换 advice.js 的生成逻辑或新增后端 AI 接口。
```

### Excel 导出

讲解点：

- 入口在 `Settings.vue`。
- 接口是 `/api/statistics/export`。
- 后端 `BillService.exportBillsExcel` 使用 Apache POI 生成 `.xlsx`。
- 前端以 blob 下载文件。

可以这样说：

```text
导出功能不是简单 CSV，而是后端生成真正的 Excel 文件，浏览器端通过文件流下载。
```

### 小程序原型

讲解点：

- 目录是 `miniapp-prototype`。
- 使用原生微信小程序结构。
- 已有登录、注册、首页、记一笔、账单、分析、我的页面。
- 请求封装在 `utils/request.js`。
- 后续可以接入 `echarts-for-weixin`。

可以这样说：

```text
目前 Web 端是完整可运行项目，小程序端是可导入微信开发者工具的原型骨架，已经复用了登录、注册、首页摘要、记账、账单和统计接口，后续可以继续完善 ECharts 小程序图表和 token 鉴权。
```

## 4. 可能被问到的问题

### 为什么使用 SSM？

回答：

```text
课程要求使用 Spring、SpringMVC、MyBatis。SSM 分层清晰，Controller 负责接口，Service 负责业务，Mapper 负责 SQL，适合展示后端开发和数据库统计能力。
```

### 为什么统计放在后端做？

回答：

```text
账单数据量变大后，如果全部交给前端统计，会增加传输量和前端计算压力。后端用 SQL 的 SUM、GROUP BY 更适合做聚合统计，也符合 MyBatis 分组统计要求。
```

### 登录为什么不是 JWT？

回答：

```text
当前 Web 端运行在浏览器环境，使用 Session 能快速实现登录态，并且和 Tomcat/SSM 集成简单。小程序正式上线时，Cookie 管理不如浏览器方便，所以文档里也说明了后续可以升级为 token 鉴权。
```

### 小程序原型是否完整可发布？

回答：

```text
目前小程序原型用于展示移动端结构和接口复用方式，还不是正式发布版本。正式发布前需要 HTTPS 后端域名、微信 request 合法域名、token 鉴权和 ECharts 小程序组件。
```

### 智能建议是不是真 AI？

回答：

```text
当前是本地规则版智能建议，属于 AI 分析功能的稳定原型。它根据预算、分类排行和趋势生成建议。后续可以替换为大模型接口，让建议更自然。
```

### 系统如何保证用户只能看自己的账单？

回答：

```text
登录后后端从 Session 获取当前用户，查询账单时把 userId 放入 BillQuery，SQL 中使用 b.user_id = 当前用户 ID 作为条件，因此不同用户数据隔离。
```

### 图表为空怎么办？

回答：

```text
先确认该月份或年份是否有账单数据，再检查统计接口是否返回 categoryStatistics 或 monthlyStatistics。前端已加入空状态提示，避免页面直接显示空白。
```

## 5. 代码讲解路线

如果老师要求现场讲代码，可以按下面路线：

1. `AuthController.java`：讲登录和 Session。
2. `AuthInterceptor.java`：讲接口拦截和 401。
3. `BillController.java`：讲账单 CRUD。
4. `BillMapper.xml`：讲动态查询、分类统计、月度统计 SQL。
5. `Dashboard.vue`：讲首页如何调用接口和渲染图表。
6. `MainLayout.vue`：讲全局快捷记账。
7. `Settings.vue`：讲分类管理、导出和深色模式。
8. `miniapp-prototype/pages/bill-form/index.js`：讲小程序如何复用后端接口。
